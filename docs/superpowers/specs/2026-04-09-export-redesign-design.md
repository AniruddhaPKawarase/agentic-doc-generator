# Export Document Redesign — Design Spec

**Date:** 2026-04-09
**Status:** Draft
**Scope:** Scope-gap pipeline export cleanup, 100% coverage fix, project name, S3 links

---

## 1. Problem Statement

The scope-gap pipeline generates export documents (Word, PDF, CSV, JSON) containing internal pipeline details not relevant to end users: completeness scores, confidence percentages, source snippets, ambiguities, gotchas, and pipeline branding. The documents also lack project identity (no project name) and source drawing references (no links to original PDFs).

Additionally, the pipeline achieves only ~76% drawing coverage on some projects, with data being silently missed.

**Constraints:**
- UI remains unchanged — all current data continues to display in Streamlit
- Only exported file content changes
- JSON export stays as full data dump (developer/integration use)

---

## 3. Architecture Overview

Four independent work streams that can be implemented in parallel:

```
Stream A: Export cleanup (document_agent.py)
Stream B: Project name from SQL (orchestrator.py + document_agent.py)
Stream C: S3 drawing hyperlinks (data_fetcher.py + orchestrator.py + document_agent.py)
Stream D: 100% coverage fix (extraction_agent.py + completeness_agent.py + orchestrator.py + config.py)
```

All streams converge in `document_agent.py` which receives enriched data and renders clean exports.

## 5. Stream B: Project Name from SQL

### 5.1 SQL Query

```sql
SELECT ProjectName, proj_address, City
FROM Projects
WHERE ProjectID = ?
```

**Database:** `IFBIMIntegration_1`
**Connection:** pyodbc, ODBC Driver 17 for SQL Server
**Credentials:** Already configured in `.env` on sandbox

### 5.2 Integration Point

The existing `services/sql_service.py` has SQL connection logic. The scope pipeline orchestrator needs to call it at pipeline start.

### 5.3 Data Flow

```
orchestrator.run()
  → sql_service.get_project_display_info(project_id)
  → returns: {"name": "SINGH RESIDENCE", "location": "Nashville, TN"}
  → passed to document_agent.generate_all(project_name=..., project_location=...)
```

**Fallback:** If SQL fails, use `"Project {project_id}"` for name, empty string for location.

**Note:** Location comes from the same SQL query at zero extra cost — `City` column on the Projects table. If City is null/empty, the Location line is omitted from the header.

### 5.4 Filename Format

```
{project_id}_{ProjectName_slug}_{Trade}_Scope_of_Work.{ext}
```

Slug rules: spaces → underscores, strip special chars, title case.

Examples:
- `7276_Singh_Residence_Concrete_Scope_of_Work.docx`
- `7276_Singh_Residence_Concrete_Scope_of_Work.pdf`
- `7276_Singh_Residence_Concrete_Scope_of_Work.csv`
- `7276_Singh_Residence_Concrete_Scope_of_Work.json`

### 5.5 Files Modified

- `scope_pipeline/orchestrator.py` — call SQL service at pipeline start
- `scope_pipeline/services/document_agent.py` — use project name in header + filename
- `services/sql_service.py` — add `get_project_display_info()` method if not present

### 5.6 Caching

Project name cached for 1 hour (already implemented in sql_service). No repeated SQL calls for the same project within the cache window.

## 6. Stream C: S3 Drawing Hyperlinks

### 6.1 Data Source

The upstream `byTrade` API returns per-record:
```json
{
  "s3BucketPath": "IFBIMIntegration_1/SinghResidence123456789",
  "pdfName": "0312202510553768A75000TYPICALDETAILSWALLSRev01-1",
  "drawingName": "A-23"
}
```

The `summaryByTrade` API may also return `s3BucketPath` and `pdfName` (project-dependent).

### 6.2 S3 URL Construction

```python
s3_url = f"https://{bucket}.s3.{region}.amazonaws.com/{s3BucketPath}/{pdfName}.pdf"
```

Or generate a presigned URL via boto3 (1-hour expiration, matching the existing document download presigned URL TTL).

### 6.3 Data Flow

```
DataFetcher.fetch_records()
  → for each record, extract: drawingName → (s3BucketPath, pdfName)
  → build dict: {"A-23": "s3://bucket/path/file.pdf", ...}
  → return as part of fetch result

orchestrator.run()
  → receives drawing_s3_urls from fetch result
  → passes to document_agent.generate_all(drawing_s3_urls=...)

document_agent._build_word()
  → for each drawing heading, if URL exists → add hyperlink
```

### 6.4 Hyperlink in Word

Using `python-docx`, add a hyperlink run on the drawing heading paragraph:
```python
# Drawing heading becomes a clickable hyperlink
_add_hyperlink(paragraph, url, drawing_name, color="#003366")
```

### 6.5 Hyperlink in PDF

The PDF builder uses `reportlab` or similar. Add URL annotation on the drawing heading text.

### 6.6 Fallback

If no S3 path available for a drawing → plain text heading (no hyperlink). No error.

### 6.7 Files Modified

- `scope_pipeline/services/data_fetcher.py` — extract S3 path mapping from records
- `scope_pipeline/orchestrator.py` — pass mapping to document agent
- `scope_pipeline/services/document_agent.py` — render hyperlinks on drawing headings

## 7. Stream D: 100% Data Coverage Fix

### 7.1 Problem

For 7276 Concrete (131 records), the pipeline extracts scope items from only 76.5% of drawings. Missing: A-13, A-24, A-25, A-5. These drawings have records in the API but the extraction agent didn't produce scope items from them.

Root causes:
1. LLM processes all records in one batch — can miss drawings with fewer records
2. Max 3 retries may not be enough for complex projects
3. CSI coverage counts irrelevant codes (Masonry, Plumbing in a Concrete report)

### 7.2 Fix 1: Increase Max Attempts

**File:** `scope_pipeline/config.py`

```python
# Change:
max_attempts: int = 3
# To:
max_attempts: int = 5
```

### 7.3 Fix 2: Batch Extraction by Drawing

**File:** `scope_pipeline/agents/extraction_agent.py`

Currently: all records for a trade sent to LLM in one call.
Change: group records by `drawing_name`, process in batches of ~30 records per LLM call.

```python
# Pseudocode
grouped = group_records_by_drawing(records)
batches = create_batches(grouped, max_records_per_batch=30)
all_items = []
for batch in batches:
    items = await self._extract_batch(batch)
    all_items.extend(items)
return all_items
```

This ensures every drawing gets dedicated LLM attention. A drawing with 3 records won't be lost among 128 others.

### 7.4 Fix 3: Force-Extract Missing Drawings

**File:** `scope_pipeline/orchestrator.py`

After the main pipeline + backpropagation completes, if `completeness.missing_drawings` is non-empty:

```python
# After final completeness check
if completeness.missing_drawings:
    for drawing_name in completeness.missing_drawings:
        drawing_records = [r for r in records if r["drawingName"] == drawing_name]
        if drawing_records:
            extra_items = await extraction_agent.extract(drawing_records)
            extra_classified = await classification_agent.classify(extra_items)
            final_items.extend(extra_classified)
    # Re-run completeness check
    completeness = completeness_agent.check(final_items, source_data)
```

This is a guaranteed sweep — each missing drawing processed individually.

### 7.5 Fix 4: Trade-Relevant CSI Filtering

**File:** `scope_pipeline/agents/completeness_agent.py`

Currently: all CSI codes from source data are counted, even irrelevant ones (Masonry codes in a Concrete report).

Change: filter `source_csi_codes` to only include codes whose 2-digit prefix matches the trade's primary CSI division.

Trade → CSI prefix mapping:
```python
TRADE_CSI_PREFIX = {
    "Concrete": ["03"],
    "Electrical": ["26"],
    "Plumbing": ["22"],
    "HVAC": ["23"],
    "Structural": ["05"],
    "Masonry": ["04"],
    # ... etc
}
```

### 7.6 Fix 5: Adjust Completeness Weights

**File:** `scope_pipeline/agents/completeness_agent.py`

```python
# Current weights:
overall = (drawing * 0.5) + (csi * 0.3) + (hallucination * 0.2)

# New weights (drawing coverage matters most):
overall = (drawing * 0.65) + (csi * 0.15) + (hallucination * 0.2)
```

### 7.7 Expected Impact

For 7276 Concrete:
- **Before:** 76.5% drawing, 41.7% CSI, 70.7% overall → incomplete, 4 missing drawings
- **After (estimated):** ~100% drawing (force-extract), ~90% trade-relevant CSI, ~97% overall → complete
